Working Around Agent Linkage Issues

January 2026 · 4 min read

Recently I've been running into a recurring issue where Tenable Nessus Agents randomly unlink from the Tenable Manager node. When that happens, devices quietly fall out of coverage until someone notices — which is exactly the kind of problem you want to detect quickly and fix consistently.

Originally, I was checking linkage manually by running nessuscli on endpoints and reviewing the agent status. That works, but it doesn't scale well when you're responsible for a large fleet. So I started looking for a way to automate the detection using tools I already had in place.

In my environment, PDQ Connect is our primary cloud deployment tool. It makes endpoint management straightforward, but at the time of writing it doesn't yet have a native PowerShell scanner (it's on the roadmap — just not available soon enough for what I needed). Because of that limitation, I couldn't build a dynamic group directly off the output of nessuscli, so I created a lightweight workaround.

The approach is simple: a script runs the nessuscli command, extracts the value that indicates whether the agent is linked or unlinked, and writes a marker file in the Temp directory. If the agent is linked, the script creates Linked.txt. If unlinked, it creates Unlinked.txt. From there, PDQ Connect's file scanner does the rest — it checks endpoints for the presence of those marker files, and I use that signal to populate two device groups: Linked and Unlinked.

I schedule the script to run weekly so I can quickly identify which devices have fallen out of linkage and relink them before coverage gaps linger. The best part is that this pattern is easy to evolve: once PDQ Connect adds a PowerShell scanner, I can remove the file-marker middle step and have the group evaluate linkage state directly. At that point it becomes a closed loop — detect unlinked agents automatically, drop them into the Unlinked group, and trigger the repair package to relink and restore coverage.

Sometimes you have to think a little outside the box to get the outcome you want, especially when working around feature gaps. Small automation wins like this make day-to-day operations much more reliable.

Working Around Agent Linkage Issues

Recently, I've come across an issue of Tenable Nessus Agents unlinking randomly from the Tenable Manager node. I was doing this manually by using the nessuscli to get the status of the agent to see if it was linked to the manager or not. So I got to thinking about how I can make this process automated using the tools I had readily available.

I use PDQ Connect as my cloud deployment tool. It is a great tool and makes endpoint management a piece of cake. Their roadmap shows a Powershell Scanner being implemented soon, but not soon enough for me. If I wanted to create a group that used the results from the nessuscli command to see if the agents are linked, I'd have to create a temporary workaround until its implementation.

So I created a script that ran the nessuscli command and extracted the value that determined if the agent was linked or unlinked. From there, if the agent was linked, it would create a file called "Linked.txt" in the Temp folder. Likewise, if the agent was unlinked, it would create a file called "Unlinked.txt". From there, I created a file scanner in PDQ Connect that looked in the Temp folders of all devices for the names of those txt files. Then I created corresponding groups so I can see devices that are linked and unlinked. I have the script scheduled to run once a week so I can see what devices have disconnected, so I can go back and link it.

Eventually, when the powershell scanner is added, I can modify the group to where if a device unlinks, it will be added to the unlinked group automatically, then the Agent Linkage repair package we have can go in and relink the agent. Sometimes you have to think outside the box in order to get the results you are looking for.
